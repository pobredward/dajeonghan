# 다정한(dajeonghan) - 단계별 개발 가이드

> **순차적으로 실행 가능한 프롬프트 모음**  
> 아무것도 없는 폴더에서 시작하여 완성된 앱까지

## 📖 이 프롬프트 모음에 대하여

이 프롬프트 모음은 **의존성 순서**와 **기능 복잡도**를 고려하여 재구성되었습니다.
각 단계는 이전 단계의 결과물을 기반으로 하며, 한 번에 하나씩 맡겨서 오류를 최소화할 수 있도록 설계되었습니다.

### 사용 방법
1. **순서대로 진행**: Step 01부터 시작하여 순차적으로 진행
2. **단계별 검증**: 각 단계 완료 후 테스트 실행 및 기능 확인
3. **오류 즉시 해결**: 다음 단계로 넘어가기 전에 현재 단계 완벽히 완료

### 예상 개발 기간
- **Full-time 개발자**: 2-3주 (MVP)
- **Part-time 개발자**: 4-6주 (MVP)
- **팀 개발 (2-3명)**: 1-2주 (MVP)

---

## 🏗️ 개발 단계 구조

### Phase 1: 기초 설정 (Foundation)
**의존성**: 없음  
**목표**: 프로젝트 뼈대 및 데이터 구조 확립

```
Step 01 → Step 02 → Step 03
(환경)    (데이터)   (엔진)
```

- **Step 01**: 프로젝트 초기 설정
  - Expo 프로젝트 생성
  - Firebase 연동
  - 필수 패키지 설치
  - 폴더 구조 생성
  - **검증**: `npx expo start` 실행 성공

- **Step 02**: 데이터 모델 정의
  - TypeScript 타입 정의
  - Firestore 스키마 설계
  - 데이터 변환 유틸리티
  - **검증**: 타입 오류 없음, Firestore CRUD 작동

- **Step 03**: 공통 엔진 구현
  - RecurrenceEngine (주기 계산)
  - PostponeEngine (미루기 패턴)
  - PriorityCalculator (우선순위)
  - NotificationOrchestrator (알림 조율)
  - **검증**: Jest 단위 테스트 통과

---

### Phase 2: 기능 모듈 (Features)
**의존성**: Phase 1 완료 필수  
**목표**: 청소, 냉장고, 약 핵심 기능 구현

```
        Step 03 (공통 엔진)
             ↓
    ┌────────┼────────┐
    ↓        ↓        ↓
Step 04  Step 05  Step 06
(청소)   (냉장고)   (약)
```

- **Step 04**: 청소 모듈
  - 방별 청소 관리
  - 10분 코스 알고리즘
  - 더러움 점수 계산
  - **검증**: 청소 테스크 생성/완료/미루기 작동

- **Step 05**: 냉장고 모듈
  - 식재료 CRUD
  - 유통기한 자동 계산
  - 임박 알림
  - 300+ 식재료 DB
  - **검증**: 식재료 추가 후 유통기한 계산 확인

- **Step 06**: 약 모듈
  - 약 정보 관리
  - 복용 스케줄
  - 복용 기록
  - 로컬 암호화
  - **검증**: 약 복용 알림 정확한 시간 발송

---

### Phase 3: 사용자 경험 (Experience)
**의존성**: Phase 2 완료 권장 (Phase 1만으로도 시작 가능)  
**목표**: 온보딩부터 일상 사용까지 완성

```
Step 04/05/06 (기능 모듈)
        ↓
    Step 07 (온보딩)
        ↓
    Step 08 (알림)
        ↓
    Step 09 (UI/UX)
```

- **Step 07**: 온보딩 시스템
  - 페르소나 선택
  - 환경 설정 질문
  - 템플릿 기반 일정 생성
  - **검증**: 온보딩 완료 후 "오늘 할 일" 즉시 표시

- **Step 08**: 알림 시스템
  - 로컬 알림 설정
  - 다이제스트 생성 (오전/저녁)
  - 약 정확한 시간 알림
  - 알림 피로 관리
  - **검증**: 다이제스트 알림 예약 확인

- **Step 09**: UI/UX 구현
  - 디자인 시스템
  - 공통 컴포넌트
  - 홈 화면
  - 네비게이션
  - **검증**: 홈 화면 → 모듈 화면 이동, 60fps 유지

---

### Phase 4: 인프라 (Infrastructure)
**의존성**: Phase 3 완료 권장  
**목표**: 프로덕션 환경 준비

```
Step 09 (UI/UX)
        ↓
    Step 10 (Firebase)
        ↓
    Step 11 (개인정보)
```

- **Step 10**: Firebase 설정
  - Security Rules 배포
  - Firestore 인덱스
  - 익명 인증
  - 오프라인 지속성
  - **검증**: Security Rules 테스트 통과

- **Step 11**: 개인정보 및 법적 준수
  - 개인정보처리방침
  - 이용약관
  - 건강 앱 면책
  - 계정 삭제 기능
  - **검증**: 개인정보처리방침 URL 접근 가능

---

### Phase 5: 출시 (Launch)
**의존성**: Phase 4 완료 필수  
**목표**: 스토어 제출 및 배포

```
Step 11 (개인정보)
        ↓
    Step 12 (성장 전략)
        ↓
    Step 14 (템플릿 마켓플레이스)  ← 신규
        ↓
    Step 13 (배포)
```

- **Step 12**: 성장 전략 및 수익화
  - 습관화 스트릭
  - 주간 리포트
  - 템플릿 공유
  - 프리미엄 게이트
  - **검증**: Analytics 이벤트 로깅 확인

- **Step 13**: 테스트 및 배포
  - 단위/통합 테스트
  - EAS Build
  - 스토어 메타데이터
  - 제출
  - **검증**: 스토어 "심사 중" 상태

## 🎯 핵심 원칙

### 1. 순차적 진행
- 이전 단계 완료 없이 다음 단계 진행 금지
- 각 단계의 완료 기준을 반드시 충족

### 2. 단계별 검증
- 코드 생성 → 테스트 → 실행 → 확인 → 커밋
- 문제 발견 시 즉시 해결 후 진행

### 3. 의존성 이해
- 각 단계가 어떤 이전 단계에 의존하는지 파악
- Phase 1은 모든 것의 기초
- Phase 2는 병렬 진행 가능 (04, 05, 06)

### 4. 오류 최소화
- 한 번에 하나의 큰 변경만 수행
- 복잡한 기능은 작은 단위로 나누어 구현
- 각 단계 완료 후 Git 커밋으로 롤백 지점 확보

## 🐛 문제 해결 전략

### 오류 발생 시
1. **현재 단계 재확인**: 프롬프트 내용 다시 읽기
2. **이전 단계 검증**: 이전 단계 완료 기준 충족 확인
3. **로그 확인**: 터미널 오류 메시지 자세히 읽기
4. **테스트 실행**: `npm test`로 어느 부분이 문제인지 파악
5. **단계별 디버깅**: 문제가 되는 파일만 집중 수정

### 막힐 때
- 각 프롬프트 파일 하단의 "문제 해결" 섹션 참고
- Firebase Emulator로 로컬 테스트
- TypeScript 타입 오류는 `any` 대신 정확한 타입 지정
